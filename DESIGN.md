# fang-image-processing: Design Doc

## 1. Overview

`fang-image-processing` lets a user upload an image and extract a sprite from it by clearing the background, using a simplified GIMP/Photoshop-style editor in the browser.

The UI follows GIMP's own layout, not just its visual style: a left sidebar stacked into three sections, top to bottom — a toolbox (tool icons), that tool's options (the same panel, options change based on the active tool, not a separate floating dialog), and a Functions row of non-tool operations (Select All, Deselect All, Invert, Remove Holes, Grow Selection, Select Border, Clear Selection, Undo, Redo, Download) laid out as a flex-wrapping button list to stay compact — the canvas fills the rest of the window, and a status bar runs along the bottom. See §5 for layout/keybinding details.

Core tools (v1):

- **Fuzzy select** — magic-wand style selection: click a pixel, select all connected pixels within a color-similarity tolerance (flood-fill). Shift+click adds an additional fuzzy-select region to the current selection instead of replacing it (union) — this is the same operation previously exposed as a separate "combine select" tool; it is now a modifier on fuzzy select, not its own toolbar entry. Ctrl/Cmd+click instead subtracts the clicked region from the current selection (the newly flood-filled region is removed from whatever was already selected) — the counterpart to Shift+click's union; Shift and Ctrl/Cmd are mutually exclusive modifiers, and with neither held (or with no existing selection) a plain click replaces the selection as normal. Keyboard shortcut: `U`. Its tool option, Threshold (the color-similarity tolerance), is a decimal value (0–128, one decimal place) shown in the toolbox's tool-options panel with a stepper input (+/- buttons, 0.1 per click) and a linked slider; the value persists across page loads via `localStorage` (client-side UI preference, not selection state — see §3 for why that split matters).
- **Gradient select** — like Fuzzy Select, but each step of the flood-fill compares a candidate pixel against the already-selected neighbor pixel it was reached from, not against the originally clicked pixel. This lets the selection walk gradually across a smooth gradient (each step within tolerance of the last, so total drift from the seed can exceed tolerance) while still stopping at a hard edge, e.g. clicking a pixel valued 147 with threshold 3 will pull in a neighboring 144 (diff 3 from the seed), then a further 141 next to *that* pixel (diff 3 from 144, the pixel it's adjacent to) — Fuzzy Select would stop after the first step since 141 has a diff of 6 from the original seed of 147. Same click/modifier conventions as Fuzzy Select (plain click replaces, Shift+click unions, Ctrl/Cmd+click subtracts) and shares its Threshold tool option. Keyboard shortcut: `G`. Already-deleted pixels (from a prior Clear) are a hard boundary the walk never crosses — see §4 for why this matters specifically for gradient select's neighbor-chained tolerance and not the other tools.
- **Select by color** — GIMP's Select > By Color: click a pixel, select every pixel in the *entire image* within the Threshold tolerance of that color, regardless of whether it's contiguous with the clicked point — unlike Fuzzy Select, which only reaches pixels connected to the seed via flood-fill. This is the tool for distributed/scattered objects (e.g. stray hairs against a background) where the matching color appears in many disconnected patches: one click grabs all of them instead of needing a Shift+click per patch. Same click/modifier conventions as Fuzzy Select (plain click replaces, Shift+click unions, Ctrl/Cmd+click subtracts) and shares its Threshold tool option — the two tools read the same persisted value, so switching between them keeps the same tolerance. Keyboard shortcut: `O`.
- **Free select** — GIMP-style lasso/polygon select: click to place path vertices; the first vertex renders as an enlarged handle (a fixed pixel radius, independent of zoom) specifically so it's easy to click back on to close the path. Clicking within that handle's radius (with at least 3 vertices placed) closes the polygon and submits it as a selection, replacing the current selection — same round-trip-to-close interaction as GIMP's free select tool. Shift+click on that closing click instead unions the polygon into the current selection, the same modifier convention as fuzzy select's add-to-selection; Ctrl/Cmd+click on the closing click instead subtracts the polygon from the current selection, the same subtract convention as fuzzy select. `Escape`, or switching to a different tool mid-path, abandons the in-progress path without submitting anything. Toolbar tool, keyboard shortcut `F`.
- **Line select** — click one point, then click a second point; every pixel on the straight segment between them is added to the selection, stroked to the width of the tool's Brush Size option (so it behaves like a straight-line brush stroke, not a 1px-wide line). A live preview line (at the current Brush Size width) follows the pointer between the first and second click so the user can see what will be selected before committing. Same click/modifier conventions as the other select tools: a plain second click replaces the current selection, Shift+click on the second click unions the line into the current selection, Ctrl/Cmd+click on the second click subtracts it. `Escape`, or switching to a different tool with a first point already placed, abandons the in-progress line without submitting anything — same convention as Free Select's in-progress path. Its tool option, Brush Size (the line's stroke width in image pixels), is an integer (1–200) shown the same way as Fuzzy Select's Threshold — stepper input + linked slider — and persists across page loads via `localStorage`, the same client-side-UI-preference pattern (see §3). Toolbar tool, keyboard shortcut `L`.
- **Rectangle select** — click one corner, then click the opposite corner; every pixel in the axis-aligned rectangle between them is selected (filled, not stroked — unlike Line Select there's no Brush Size or any other width option, since the whole enclosed area is the selection). Same click/modifier conventions as the other select tools: a plain second click replaces the current selection, Shift+click on the second click unions the rectangle into the current selection, Ctrl/Cmd+click subtracts it. A live preview rectangle (dashed outline, translucent fill) follows the pointer between the first and second click, the same role Line Select's preview line plays. `Escape`, or switching to a different tool with a first corner already placed, abandons the in-progress rectangle without submitting anything — same convention as Line Select's in-progress line. Toolbar tool, keyboard shortcut `R`.
- **Brush** — freehand painting: drag to paint a stroke at the tool's Brush Size width (the same option Line Select uses); releasing the mouse commits that stroke as a selection. Painting lives **entirely in the browser** for the life of the drag — nothing is sent to Rails until mouseup, so dragging never round-trips per frame the way every click-based tool does. While dragging, the in-progress stroke is drawn as a translucent blue preview at Brush Size width (same visual language as Line Select's preview segment), with an idle brush-size cursor circle when not dragging — no full-canvas tint. Same click/modifier conventions as the other select tools, read off the releasing mouseup rather than a click: a plain release replaces the current selection with the painted stroke, Shift+release unions it into the current selection, Ctrl/Cmd+release subtracts it. `Escape`, or switching to a different tool mid-stroke, discards the in-progress painting without submitting anything — same convention as Free Select/Line Select/Split Selection's in-progress-gesture cancel. Toolbar tool, keyboard shortcut `B`.
- **Split selection** — only available when there is a current selection: click two points to set an infinite line's direction (the two points only fix direction/position, not a segment — the line always extends across the whole image), then click a third point on whichever side of that line should be kept. The current selection is replaced with whatever part of it falls on the kept side; the part on the other side of the line is discarded entirely (not just deselected — it's gone, same "replace" semantics as Select Border/Grow, not an add/subtract click-select). No add/subtract modifier, since there's nothing to union or subtract against — like Invert, this is a one-shot transform of the current selection. `Escape`, or switching tools with the line's direction points already placed, abandons the in-progress split without submitting anything, the same convention as Free Select/Line Select. Toolbar tool, keyboard shortcut `S`; the toolbar button is always visible but is a no-op (same silent-guard convention as Invert/Remove Holes/Save Selection) if there is no current selection when the third click lands.
- **Smooth Auto Fill** — only available when there is a current selection: click two points to set a direction vector (like Split Selection, the two points only fix direction, not a segment), then sweeps a family of 1px-spaced lines parallel to that vector across the whole image. Each line can cross one or more separate gaps of selected, transparent pixels; for each such gap, the tool finds the closest non-transparent pixel immediately bordering it on either side (searched from the *entire* image, not limited to the selection — the anchor colors can come from outside what's selected, e.g. background just past the selection's edge), then, from each of those bordering pixels, keeps walking further along the line away from the gap collecting further pixels that are both contiguous (no transparent gap) and within the tool's Threshold tolerance of that immediate bordering pixel's color — the same color-similarity test Fuzzy Select uses, just walked along a line instead of flood-filled across an area — and averages every pixel collected this way into that side's actual anchor color, rather than using the single bordering pixel's color verbatim. This is what keeps one outlier pixel right at a gap's edge (e.g. a stray speck of noise) from dictating the whole gap's fill color; a smoother, more representative patch of the surrounding texture is used instead. Color is then linearly interpolated across the gap between those two averaged anchors. A gap missing a bordering anchor on either side (including running off the image edge first) is left untouched entirely — there's nothing to interpolate between. This is for smoothing over a gap in a gradient/textured background (e.g. a scratch or a removed object) by blending in the direction of the surrounding pattern's grain, rather than a flat fill. Shares the Threshold tool option with Fuzzy Select/Gradient Select/Select by Color (same persisted `localStorage` value — see above), though here it gates the anchor-averaging walk rather than a flood-fill/selection test. No add/subtract modifier — like Split Selection, this doesn't change what's selected, only the pixel colors within it; unlike every other tool, it writes actual RGB values onto the canvas (see §4), not a new mask. `Escape`, or switching tools with the first direction point already placed, abandons the in-progress pick without submitting anything, the same convention as Split Selection/Line Select. Toolbar tool, keyboard shortcut `A`; the toolbar button is always visible but is a no-op (same silent-guard convention as Invert/Split Selection) if there is no current selection when the second click lands.
- **Select all** — replaces the current selection with the entire image. Triggered via `Ctrl/Cmd+A`, or the Select All button in the sidebar's Functions row (see §5). Always a fresh full-image mask; never additive.
- **Deselect all** — drops the current selection entirely (no mask). Triggered via `Ctrl/Cmd+D`, or the Deselect All button in the sidebar's Functions row (see §5). Distinct from Clear (`Delete`/`Backspace`): deselecting never touches canvas pixels, it only forgets what was selected.
- **Invert select** — flip the selection: selected becomes unselected and vice versa. Triggered via `Ctrl/Cmd+I`, or the Invert button in the sidebar's Functions row (see §5).
- **Remove holes** — GIMP's Select > Remove Holes: folds any unselected region fully enclosed by the current selection into the selection itself (e.g. a fuzzy-selected donut shape becomes a filled disc). Has no toolbar icon — it's a one-shot transform of the current selection, reached via `Ctrl/Cmd+Shift+H` or the Remove Holes button in the sidebar's Functions row (see §5), the same bucket as Select All/Deselect/Invert. A no-op (422, same as Invert) if no selection is active.
- **Grow selection** — GIMP's Select > Grow: expands the current selection outward by a configurable Border Size (in pixels), adding that expanded ring to the existing selection (the interior stays selected). Has no toolbar icon — reached via the Grow Selection button in the sidebar's Functions row (see §5), same bucket as Remove Holes. Clicking it opens a small dialog to confirm the Border Size before running; a no-op (422, same as Invert) if no selection is active.
- **Gradient select from selection** — like Gradient Select, but seeded from every pixel on the current selection's boundary instead of a single clicked point: walks outward from the whole boundary at once, each candidate pixel compared against the already-selected neighbor it was reached from (same tolerance semantics as Gradient Select), stopping wherever it hits a hard edge. Only ever adds pixels to the current selection; never shrinks it. Has no toolbar icon — reached via the Gradient Select from Selection button in the sidebar's Functions row (see §5), same bucket as Grow Selection, and reuses the Threshold value already in the tool-options panel rather than prompting for a new one. A no-op (422, same as Invert) if no selection is active. Intended workflow (see the "typical flow" paragraph below): Gradient Select stops at a hard edge; Grow Selection expands past that edge; Gradient Select from Selection then resumes the outward gradient walk from the newly-grown boundary; repeat the grow/gradient-select-from-selection cycle until the whole object is captured. Same already-deleted-pixels-are-a-hard-boundary treatment as Gradient Select — see §4.
- **Select border** — GIMP's Select > Border: replaces the current selection with just the ring of pixels straddling its boundary, at the same configurable Border Size, dropping the rest of the interior — the counterpart to Grow Selection (grow keeps the interior and adds a ring; border keeps only the ring). Has no toolbar icon — reached via the Select Border button in the sidebar's Functions row (see §5). Shares the same Border Size dialog and value as Grow Selection: both read/write one `localStorage` preference (`fang-editor:border-size`), matching GIMP's own convention of remembering a single grow/border size across the two rather than one per operation — see §5. A no-op (422, same as Invert) if no selection is active.
- **Clear selection** — clears (deletes) the currently selected pixels to transparent, updating the canvas in place. Triggered via `Delete`/`Backspace`, or the Clear Selection button in the sidebar's Functions row (see §5), GIMP's Edit > Clear equivalent. **Additive**: each clear composites onto whatever the canvas already shows, so clearing one region and then selecting and clearing a different region keeps both cleared — it does not reset to the original upload. The mask stays attached after a clear (matches GIMP: clearing doesn't deselect).
- **Save selection** — saves the current selection into one of up to 3 slots on the `EditSession`, so it can be recalled later even after making other selections. No toolbar icon — reached via the Save Selection button in the sidebar's Functions row (see §5), the same bucket as Clear Selection. A no-op (422, same as Invert) if no selection is active. Slots are not a toolbar/tool concept and have no keybinding; once all 3 slots are full, saving again **rolls over the oldest saved slot** (by `updated_at`) rather than prompting the user to pick one to discard — same "keep the N most recent" ergonomics as shell history. Saved selections are separate from undo/redo history (§3): saving one is not itself an undo step.
- **Merge saved selection** — unions the current selection with a previously saved one (GIMP has no direct equivalent; closest is loading a saved selection channel and combining). No toolbar icon — reached via the Merge Saved Selection button in the sidebar's Functions row (see §5), which opens a dialog listing the session's saved selections (thumbnail + slot number) to pick from. Picking one merges it into the current selection via the same `/combine` op the Shift+click add-to-selection modifier uses (§4); if nothing is currently selected, the saved selection simply becomes the current selection. Unlike Save Selection, this **is** a normal undo step (`record_snapshot!`), since it changes what's selected the same way any other select tool does.
- **Undo / redo** — GIMP-style history, one step per tool operation (fuzzy select, free select, Shift+click add, select all, deselect all, invert, remove holes, clear). Triggered via `Ctrl/Cmd+Z` (undo) / `Ctrl/Cmd+Shift+Z` (redo), or the Undo/Redo buttons in the sidebar's Functions row (see §5). See §3 for the snapshot model.
- **Zoom** — a single toolbar tool (not two separate in/out buttons): click zooms in, Ctrl/Cmd+click zooms out, GIMP-style, both anchored on the clicked point so that exact image pixel stays fixed under the cursor as the view rescales (see §5 for the anchor math). `+`/`-` keyboard shortcuts zoom in/out anchored on the current viewport center instead (no click point to anchor on), and `0` resets to 100%. Purely a view transform — scales the on-screen canvas rendering only, never resizes the underlying `<canvas>` elements, touches pixel data, or calls Rails/Python. The zoom level persists across page loads via `localStorage`, the same client-side-UI-preference pattern as fuzzy select's Threshold (see above) — it is not part of `EditSession`/selection state (§3) and is irrelevant to undo/redo, since it never changes what pixels exist, only how large they're drawn.
- **Download** — exports whatever the canvas currently shows: the untouched original if Clear hasn't been used yet, or the accumulated transparent cutout after one or more clears. Download never triggers a clear itself; it is a pure export of current state. Not a tool (it doesn't change what clicking the canvas does) — triggered via `Ctrl/Cmd+S`, or the Download button in the sidebar's Functions row (see §5), alongside the other non-tool functionality.

Typical flow: fuzzy-select the background → `Ctrl/Cmd+I` to invert (so the subject is now selected) → or fuzzy-select the background, `Shift+click` a few more background regions that weren't contiguous → `Delete`/`Backspace` to clear it to transparent → click Download. Either selection order works because the operations compose; Clear and Download are always the last two steps, in that order. Any step can be undone/redone afterward, including after multiple clears — see §3. Free select composes the same way as fuzzy select (plain replaces, Shift+click-to-close unions), so it drops into the same flow wherever a region isn't easily reachable by flood-fill tolerance alone. Gradient select is a drop-in swap for fuzzy select in this flow when the background is a smooth gradient/vignette rather than a flat color — it reaches further across the gradient than fuzzy select would at the same threshold, without needing a threshold so high it also eats into the subject. For a gradient/vignette that's too wide for a single gradient select to cross even one hard edge boundary at a time, combine it with Gradient Select from Selection and Grow Selection: Gradient Select from a seed point stops at the first hard edge it can't cross; Grow Selection expands the selection past that edge by a few pixels; Gradient Select from Selection then resumes the outward gradient walk from there; repeating that grow/gradient-select-from-selection cycle steps the selection across each successive hard edge until the whole object is covered. `Ctrl/Cmd+A`/`Ctrl/Cmd+D` are quick resets — e.g. Select All then Clear to wipe the canvas entirely, or Deselect All to back out of a selection without undoing the tool operation that made it.

**Non-goals for v1**: not a general paint/photo editor. No layers (the Brush tool above is a selection input method, not a paint/drawing tool — it never touches canvas pixels, only produces a mask), no multi-image batch processing, no ML-based one-click background removal (may be added later via the Python service, see §7). Undo/redo history persists only for the life of the `EditSession` row (no explicit expiry/cleanup job yet), not indefinitely across unrelated sessions.

**Output contract**: the final export is a transparent PNG (RGBA), same pixel dimensions as the source image. This matches `fang-backend`'s `Sprite.url` expectation (a pre-made transparent PNG/WebP file) even though there is no live integration today — see §7.

This app is standalone: no API calls to `fang-backend`. The user downloads the resulting PNG directly.

**Upload and session management**: available identically on the landing page and the editor page (see §5's "Session management UI" for the shared controller behind this). Dragging an image file anywhere onto either page, or `Ctrl/Cmd+Shift+O` (opens the native file picker), uploads it and always creates and navigates to a **new** `EditSession` — whatever session is currently open, if any, is left completely untouched in the database; this is additive, never a replace-in-place. `Ctrl/Cmd+O` opens a Load Session dialog listing every existing `EditSession` (thumbnail + created date) to click through to, with a delete action (behind a confirmation prompt) per row. There is no per-user scoping today — the same "no user accounts" model as `session_token` (see §3) — so the list is simply every session in the database, not filtered to "sessions I created."

## 2. Architecture

```
Browser (Stimulus + Canvas API)
  - renders uploaded image on <canvas>
  - renders the selection mask returned by Rails as a visual overlay
  - user clicks/drags -> POST coordinates + tool + params to Rails
  - never computes pixels itself; canvas is for display + input capture only

Rails (app + models + services)
  - stores the original upload (Active Storage, local disk service)
  - owns the authoritative EditSession + selection mask (server-side state)
  - proxies pixel operations to the Python service via PixelEngineClient
  - returns the updated mask (as a PNG) to the browser after every operation

Python microservice (FastAPI, local process, separate runtime)
  - stateless: receives image + current mask + op + params, returns a new mask
  - fuzzy select: OpenCV flood-fill / connected-component labeling with a
    color-distance tolerance
  - select by color: like fuzzy select's tolerance test, but applied to every
    pixel in the image independently (no flood-fill/connectivity) — matches
    scattered, non-contiguous regions in one pass
  - gradient select: custom BFS flood-fill, tolerance compared against the
    adjacent selected pixel rather than the seed pixel; already-deleted
    pixels (alpha 0) are excluded from the walk so it can't bridge through
    a cleared region into fresh pixels beyond it
  - gradient select from selection: same BFS, seeded from the whole current
    selection boundary instead of a single point
  - combine: boolean OR of two masks
  - invert: boolean NOT of a mask
  - delete: apply the mask as the alpha channel, return the final transparent PNG
  - smooth auto fill: sweep 1px-spaced lines parallel to a given direction
    vector across the image; per gap of selected/transparent pixels on a
    line, average each side's run of color-similar, contiguous pixels
    walking outward from the closest non-transparent pixel immediately
    bordering that gap (a tolerance-gated walk, same test as fuzzy select,
    to smooth over outlier anchor pixels), then linearly interpolate color
    across the gap between those two averaged anchors — writes pixel
    colors, not a mask, like delete
```

Why a Python microservice instead of doing pixel math in Ruby: OpenCV/Pillow (and optionally `rembg` later) give access to a much broader, more battle-tested image-processing ecosystem than is available in Ruby. Rails owns the app/UI/session/upload lifecycle; Python owns pixel math exclusively. The two communicate over local HTTP through a single Rails-side client class — see §5.

## 3. Selection state model

The selection mask is a single-channel bitmap (0 = unselected, 255 = selected), the same width/height as the source image. It is **stored server-side**, keyed by an `EditSession`, not computed or held authoritatively in the browser.

Why state is needed at all: unlike a stateless single-shot filter, fuzzy-select/combine/invert/delete are inherently compositional — "combine" and "invert" both operate on whatever is *currently* selected, so there has to be a canonical "current selection" somewhere between requests.

Why the browser doesn't own that ground truth: after every tool operation, Rails returns the updated mask (rendered as a PNG overlay) and the browser simply redraws it. This keeps client and server from ever drifting out of sync — the browser is a thin renderer of server state, not a second copy of the truth.

Two things have to hold for "canonical current selection" to actually be true rather than just asserted, given every tool action is a read-Python-write sequence (read `current_mask`, call the Python service, write a new snapshot) with no atomicity across that sequence on its own:

- **Same-session requests must not interleave server-side.** Two Shift+click (add-to-selection) requests fired in quick succession are handled on separate Puma threads; without serialization, request B can read `current_mask` before request A's snapshot commits, so B's write silently clobbers A's (e.g. selecting regions 1, 2, 3 in quick succession ends up with only 1+3, region 2's snapshot briefly existing then getting overwritten). `EditSessions::SelectionsController` serializes same-session requests with a per-`EditSession` in-process `Mutex` (`lock_edit_session`, an `around_action`) rather than `ActiveRecord#with_lock`/DB-level locking, since SQLite (this app's adapter) has no row-level `FOR UPDATE` support and its single-writer lock is only acquired at the first actual write, too late to prevent the read side of the race.
- **The browser must not fire requests out of the order the user triggered them.** Every tool action (`click()`, `clearSelection()`, `invert()`, etc.) is an independent async handler with no shared in-flight tracking on its own — selecting a region and immediately pressing Delete/Backspace before the select's response arrives fires two overlapping fetches with no guarantee the server receives them in trigger order, so Delete can run against the mask from *before* the select, deleting the wrong region even though the server handled each request correctly in isolation. `editor_controller.js#post()` (the single chokepoint every tool action's request goes through) chains each call behind `this.requestQueue`, a promise that resolves once the previous request's response has been received, so a request is never sent until every previously-queued one has finished.

`EditSession` (Rails model):
- `has_one_attached :original_image`
- `has_one_attached :current_mask` — always mirrors whatever `EditSessionSnapshot` the session is currently positioned at (see below); not written to directly by controllers.
- `has_one_attached :canvas_image` — the accumulating visual result of clears applied so far. `display_image` returns this if attached, otherwise `original_image` (no clear has happened yet). **Additive**: `delete` reads `display_image` (not `original_image`) as the base to composite the current mask onto, so a second clear layers onto the first instead of overwriting it back to the untouched upload. `fuzzy_select` and `free_select` also read `display_image` (not `original_image`) for the same reason: flood-fill/rasterization must see the canvas as it currently looks (post-delete transparent regions included), not the pristine upload, or a fuzzy-select after a delete would flood-fill against original colors that were never actually cleared from that base image and behave as if nothing had been deleted. Also mirrors the current snapshot, same as `current_mask`.
- `current_snapshot_position:integer` (default `-1`) — pointer into this session's `snapshots`, where `-1` means "before the first operation" (no mask, no canvas — just `original_image`).
- `has_many :snapshots, class_name: "EditSessionSnapshot"` — the undo/redo history; see below.
- `session_token:string` (unique, generated on create) — identifies "who is editing" for v1; no user accounts. Not currently read from a cookie/param (each `EditSession` is reached directly by its record id), but the column exists so per-browser scoping can be added without a schema change.

### Undo/redo (`EditSessionSnapshot`)

Every tool operation (`fuzzy_select` including Shift+click add, `invert`, `delete`) appends one `EditSessionSnapshot` row and advances `current_snapshot_position` to it — one undo step per operation, GIMP-style. Each snapshot is self-contained (not a diff): it carries `mask` (nullable Active Storage attachment) and `canvas_image` (nullable Active Storage attachment), so restoring a position is a pure copy with no replay needed.

- `position:integer` — 0-based, unique per `edit_session_id`.
- `operation:string` — which tool produced this step (`"fuzzy_select"`, `"invert"`, `"delete"`), for debugging/inspection only; undo/redo doesn't branch on it.
- `mask` — the selection mask after this operation. Carried forward unchanged on `delete` (clearing doesn't deselect).
- `canvas_image` — the composited canvas after this operation. For `fuzzy_select`/`invert`, which don't touch pixels, this is **carried forward from the previous snapshot's canvas blob** (same Active Storage blob, not a duplicate write) rather than left blank — otherwise undoing past a `delete` to an intervening selection step would find no canvas on that snapshot and incorrectly revert the visible image all the way to `original_image` instead of showing it as it looked at that point in history.

`EditSession#record_snapshot!(operation:, mask_bytes:, canvas_bytes: nil)` is the only way snapshots get created. It first destroys any snapshots beyond `current_snapshot_position` (the standard undo/redo rule: a new edit after an undo discards the redo branch you undid past), then creates the new snapshot and copies its blobs onto the live `current_mask`/`canvas_image` attachments via blob-sharing `attach` calls (no byte duplication between a snapshot and the session's live attachments).

`EditSession#undo!` / `#redo!` move `current_snapshot_position` by one and re-copy that position's blobs onto `current_mask`/`canvas_image` (or purge both if moving to position `-1`). Both return `false` (no-op) at the ends of history — no snapshot to move to — which the controller surfaces as a 422 (`render_no_history`, see §5).

### Saved selections (`EditSessionSavedSelection`)

Separate from undo/redo: a small, user-controlled set of selections the user explicitly opts to keep around (via Save Selection, §1) so they can be recalled later regardless of how many other operations happened in between — `snapshots` above is automatic and linear, this is manual and slot-based.

`EditSession#has_many :saved_selections, class_name: "EditSessionSavedSelection"` — each row is `slot:integer` (0..`MAX_SAVED_SELECTIONS - 1`, unique per `edit_session_id`) plus a `mask` attachment (single-channel PNG, same 0/255 convention as `current_mask`). `MAX_SAVED_SELECTIONS = 3`.

`EditSession#save_selection!` picks a slot via `next_saved_selection_slot`: the first empty slot if fewer than `MAX_SAVED_SELECTIONS` are used, otherwise the slot belonging to the saved selection with the oldest `updated_at` (rolling it over) — this is what makes saving always succeed without asking the user which of the 3 to discard. `EditSession#merge_selection!(saved_selection:, pixel_engine:)` unions `current_mask` with the saved selection's mask via `PixelEngineClient#combine` (falling back to just the saved mask if nothing is currently selected) and calls `record_snapshot!` with the result — merging *is* a normal undo step, unlike saving.

## 4. Rails ↔ Python contract

Internal HTTP API, not public-facing. Wire format: JSON bodies with images/masks as base64-encoded PNG bytes (chosen over multipart for simplicity — every payload is small, local-only, and already JSON-shaped alongside scalar params like `x`/`y`/`tolerance`). The contract itself is treated as versioned and stable so the two sides can be worked on independently.

| Endpoint | Request | Response |
|---|---|---|
| `POST /fuzzy_select` | `{image, mask?, x, y, tolerance}` (base64 PNG + ints) | `{mask}` (base64 single-channel PNG, 0/255) |
| `POST /gradient_select` | `{image, mask?, x, y, tolerance}` (base64 PNG + ints, same shape as `fuzzy_select`) | `{mask}` (base64 single-channel PNG, 0/255) |
| `POST /select_by_color` | `{image, mask?, x, y, tolerance}` (base64 PNG + ints, same shape as `fuzzy_select`) | `{mask}` (base64 single-channel PNG, 0/255 — every matching pixel in the image, not just the connected region) |
| `POST /gradient_select_from_selection` | `{image, mask, tolerance}` (base64 PNGs + int; `mask` required, not optional) | `{mask}` (base64 single-channel PNG, 0/255 — current selection grown outward) |
| `POST /select_all` | `{image}` (base64 PNG) | `{mask}` (fully 255, same dimensions as `image`) |
| `POST /free_select` | `{image, points}` (base64 PNG + list of `[x, y]` ints) | `{mask}` (polygon rasterized via scanline fill) |
| `POST /line_select` | `{image, x1, y1, x2, y2, brush_size}` (base64 PNG + ints) | `{mask}` (line segment stroked to `brush_size` width) |
| `POST /rect_select` | `{image, x1, y1, x2, y2}` (base64 PNG + ints, same shape as `line_select` minus `brush_size`) | `{mask}` (axis-aligned rectangle between the two corners, filled) |
| `POST /brush_select` | `{image, strokes, brush_size}` (base64 PNG + list of point-lists + int) | `{mask}` (every stroke's polyline stroked to `brush_size` width, unioned together) |
| `POST /split_selection` | `{mask, x1, y1, x2, y2, keep_x, keep_y}` (base64 PNG + ints) | `{mask}` (current selection intersected with the half-plane, on the `(x1,y1)`–`(x2,y2)` line, containing `(keep_x, keep_y)`) |
| `POST /smooth_auto_fill` | `{image, mask, x1, y1, x2, y2, tolerance}` (base64 PNGs + ints; `image` is the current canvas with alpha, `mask` is the current selection, `(x1,y1)`–`(x2,y2)` set a direction vector, `tolerance` gates the per-side anchor-averaging walk, same shape/default as `fuzzy_select`'s) | `{png}` (base64 RGBA PNG — like `delete`, writes actual pixel colors, not a mask) |
| `POST /combine` | `{mask_a, mask_b}` (base64 PNGs) | `{mask}` |
| `POST /subtract` | `{mask_a, mask_b}` (base64 PNGs) | `{mask}` (mask_a with mask_b's selected pixels removed) |
| `POST /invert` | `{mask}` (base64 PNG) | `{mask}` |
| `POST /remove_holes` | `{mask}` (base64 PNG) | `{mask}` (unselected regions enclosed by selection folded in) |
| `POST /grow_selection` | `{mask, border_size}` (base64 PNG + int) | `{mask}` (selection dilated outward by `border_size` pixels) |
| `POST /select_border` | `{mask, border_size}` (base64 PNG + int) | `{mask}` (ring of `border_size` pixels straddling the selection boundary) |
| `POST /delete` | `{image, mask}` (base64 PNGs) | `{png}` (base64 RGBA PNG, transparent cutout) |

`select_all` and `free_select` take `image` only to derive pixel dimensions (`image.shape` in `masks.py`) — same reasoning as every other endpoint reading dimensions off the decoded image rather than trusting caller-supplied width/height, so Rails never needs to keep a second source of truth for the image's size in sync with the upload. Deselecting entirely (Ctrl/Cmd+D) has no Python endpoint: it's a pure "drop the mask" operation with no pixel math involved, handled entirely in `EditSessions::SelectionsController#deselect_all` by recording a snapshot with no mask attached (see §3, §5).

`remove_holes` (`masks.remove_holes`) labels the connected components of the *unselected* area (`cv2.connectedComponents` on the inverted mask); any component that never touches the mask's border is fully enclosed by selection and gets OR'd into the returned mask, while border-touching unselected regions (background) are left alone. Like `invert`, it's a pure mask-in/mask-out transform with no `image` needed.

`grow_selection` (`masks.grow_selection`) dilates the mask with a single square structuring element sized `border_size * 2 + 1`, which expands the selection by exactly `border_size` pixels in every direction — dilation alone already produces the union of the original selection and the new ring, so no separate combine step is needed. `select_border` (`masks.select_border`) dilates and erodes by half `border_size` each and takes the difference (`dilated AND NOT eroded`), which centers the resulting ring on the original boundary rather than falling entirely outside or inside it, matching GIMP's own Select > Border behavior. Both are pure mask-in/mask-out transforms with no `image` needed, same shape as `invert`/`remove_holes`.

Masks follow the convention in §3: 0 = unselected, 255 = selected. `delete` ANDs `NOT(mask)` with whatever alpha `image` already has (`masks.decode_image_with_alpha`/`apply_mask_as_alpha`) rather than discarding existing transparency and rebuilding alpha from the mask alone — a pixel comes out transparent if it was already transparent *or* is newly selected. This is what makes additive delete actually additive at the pixel level: `fuzzy_select`/`invert`/`combine` decode with `decode_image` (RGB only, alpha irrelevant to flood-fill), but `delete` must decode with `decode_image_with_alpha` or a second delete on a fresh selection would silently un-clear every pixel cleared by the first one. The Python service itself is still a one-shot, stateless composite (it has no notion of "additive" or history) — `image` here is whatever Rails passes as the base to composite onto. The Rails side supplies which base image: `EditSessions::SelectionsController#delete` passes `@edit_session.display_image` (the accumulated canvas so far, see §3), not always `original_image`, so successive clears layer onto each other instead of each restarting from the untouched upload.

`apply_mask_as_alpha` also zeroes RGB (not just alpha) wherever the resulting alpha is 0, so every transparent pixel is uniformly `#00000000` rather than alpha-zero-but-still-carrying-its-original-RGB. This matters because Pillow's `Image.convert("RGB")` (used by `decode_image`) discards alpha without zeroing the underlying color — a transparent pixel keeps whatever RGB it had before deletion. Without the zeroing step, "select all → delete → fuzzy-select again" would flood-fill against non-uniform stale RGB values under the transparent region and fail to select all of it; zeroing makes the deleted area color-uniform so a later fuzzy select on any transparent pixel selects the whole transparent region.

The Python service's endpoint set is unchanged from v1 — `/combine` still exists and is still used internally, just no longer surfaced as its own Rails route/toolbar tool. `/subtract` (`masks.subtract`) is the newer counterpart: it ANDs `mask_a` with `NOT(mask_b)`, i.e. removes `mask_b`'s selected pixels from `mask_a`, the mask-level operation behind Ctrl/Cmd+click's "subtract from selection" modifier.

`select_by_color` (`masks.select_by_color`) takes the same request shape as `fuzzy_select` — `{image, mask?, x, y, tolerance}` — but tests every pixel in the image independently against the clicked seed color (`np.abs(image - seed).max(axis=2) <= tolerance`) rather than flood-filling outward from the seed. This is what makes it match scattered/non-contiguous regions in a single click, unlike `fuzzy_select`'s `cv2.floodFill`, which can only ever reach pixels connected to the seed. Same tolerance semantics (max per-channel absolute difference) as `fuzzy_select`/`gradient_select`, so the Threshold tool option means the same thing across all three. Like `gradient_select`, it also excludes already-deleted pixels (alpha 0) from the result — a cleared region is a uniform zeroed-RGB block (see `apply_mask_as_alpha`) that would otherwise match almost any tolerance against a dark seed color and get re-selected wholesale. Routes through the same `mask_with_modifier` add/subtract shape as `fuzzy_select`/`gradient_select` on the Rails side.

`gradient_select` (`masks.gradient_select`) takes the same request shape as `fuzzy_select` but cannot reuse `cv2.floodFill`: OpenCV's `loDiff`/`upDiff` are fixed relative to the seed pixel for the life of the fill, and gradient select instead needs the comparison reference to shift to whichever already-selected pixel a candidate was reached from. It's a plain BFS over 4-connected neighbors starting at `(x, y)`, each step comparing the candidate against the neighbor it came from (max of the per-channel absolute diff, same as `floodFill`'s tolerance semantics) rather than the original seed — see §1 for the worked example of why this reaches further across a smooth gradient than `fuzzy_select` at the same threshold while still stopping at a hard edge.

`gradient_select`/`gradient_select_from_selection` additionally derive the image's alpha channel from the same `image` bytes (`masks.decode_alpha`) and pass it into the BFS as `alpha`: any pixel with alpha 0 (already cleared by a prior Clear, per the RGB-zeroing in `apply_mask_as_alpha`) is excluded from the walk's frontier, so it can neither be re-selected nor act as a stepping stone into pixels beyond it. This is scoped narrowly to these two gradient-walk endpoints, not a general "deleted pixels are unselectable" rule — `fuzzy_select` and every mask-only op (`invert`, `grow_selection`, `select_border`, `remove_holes`, `combine`, `subtract`) are untouched, since a already-deleted region is exactly the kind of uniform, zeroed-RGB block gradient select's neighbor-chained tolerance is prone to bridge straight across at moderate thresholds, silently pulling in fresh pixels on the far side; fuzzy select's seed-relative tolerance doesn't have the same failure mode and changing what those other ops treat as selectable would ripple into their own well-defined mask semantics for no benefit.

`gradient_select_from_selection` (`masks.gradient_select_from_selection`) reuses `gradient_select`'s BFS shape but seeds the queue from every currently-selected pixel up front (`deque(zip(*np.nonzero(selected)))`) instead of a single `(x, y)` point, so the walk expands outward from the entire boundary simultaneously rather than from one origin. It mutates a copy of the incoming mask by OR-ing in newly reached pixels — the existing selection is never removed, so this is a pure "grow via gradient tolerance" op rather than a replace. Unlike `gradient_select`, `mask` is required (there's no boundary to seed from without one), which is why it's a separate endpoint rather than an optional-seed variant of the existing one.

`line_select` (`masks.line_select`) draws a single `cv2.line` stroke of width `brush_size` between `(x1, y1)` and `(x2, y2)` onto an otherwise-empty mask — GIMP's `cv2.line` thickness parameter already rasterizes a segment of arbitrary width, so this needs no custom distance-to-segment math the way `free_select`'s polygon fill or `gradient_select`'s BFS do. Like `select_all`/`free_select`, it takes `image` only to derive output dimensions, not for any pixel comparison — a straight-line selection doesn't depend on the underlying colors at all.

`rect_select` (`masks.rect_select`) is line_select's closest relative but fills rather than strokes: `cv2.rectangle` with `thickness=-1` fills the axis-aligned box between the two opposite corners `(x1, y1)`/`(x2, y2)` directly, so there's no `brush_size` param at all — the whole enclosed area is the selection, not a path through it. `cv2.rectangle` interprets any two opposite corners the same way regardless of which is top-left/bottom-right, so the corners never need min/max normalization before being passed in. Same reasoning as `line_select` for taking `image` only to derive output dimensions — a rectangular selection doesn't depend on the underlying colors either.

`brush_select` (`masks.brush_select`) generalizes `line_select` from one segment to a whole freehand path: `strokes` is a list of point-lists (one per continuous mouse-down-to-mouse-up drag — the browser currently commits one stroke per request on mouseup, see §5, but the wire format keeps strokes as a list so disconnected dabs could still be batched later without a contract change) rather than a single flattened point list, specifically so two disconnected dabs don't get joined by a spurious line drawn between the last point of one stroke and the first point of the next. Each stroke is drawn as a chain of `cv2.line` segments at `brush_size` thickness (same reasoning as `line_select` for reusing `cv2.line`'s thickness param over hand-rolled distance math), with a `cv2.circle` stamped at every vertex so joins between segments don't look faceted at large brush sizes or sharp turns; a stroke of a single point (a click with no drag) is stroked as just that one circle. Like `line_select`, it takes `image` only to derive output dimensions — a freehand selection doesn't depend on the underlying colors.

`split_selection` (`masks.split_selection`) takes no `image` at all — it's a pure mask-in/mask-out transform like `invert`/`grow_selection`, needing only dimensions already implicit in `mask`. `(x1, y1)`/`(x2, y2)` fix an infinite line's direction and position (not a segment — unlike `line_select`, the two points are never themselves selected or used as stroke endpoints); every mask pixel's coordinates are plugged into that line's implicit equation `(x - x1)*(y2 - y1) - (y - y1)*(x2 - x1)`, whose sign alone says which side of the line the pixel falls on (the standard 2D cross-product side test). `(keep_x, keep_y)` is run through the same expression to find which sign to keep; every mask pixel whose sign doesn't match is zeroed, everything else (including pixels exactly on the line, kept rather than discarded, since the line has zero width) passes through unchanged. This can only ever shrink the current selection, never grow it — the counterpart to Grow Selection in that sense, though its replace-not-add semantics are closer to Select Border/Invert.

`smooth_auto_fill` (`masks.smooth_auto_fill`) is the odd one out in this table: every other endpoint here is mask-in/mask-out (or, for `delete`, mask-plus-image-in/PNG-out with a fixed alpha-compositing rule) — this one takes a direction vector and produces new pixel *colors*, not a selection. `(x1, y1)`/`(x2, y2)` set direction/position only, the same "infinite line, not a segment" convention as `split_selection`'s two points, not `line_select`'s stroke endpoints. Every pixel in the image is bucketed into a 1px-wide line parallel to that direction (grouped by rounding its perpendicular offset from the line through `(x1, y1)` to the nearest integer — the same "which side/line does this pixel belong to" question as `split_selection`'s side test, just bucketed by distance instead of sign) and ordered within its line by its projection along the direction vector. Within each line, contiguous runs of fillable pixels (selected AND transparent) are found and filled independently — each run's two immediate bordering anchors are the non-transparent pixels *immediately bordering that run* (one step before it starts, one step after it ends along the line), not the line's global first/last non-transparent pixel; a line can cross other opaque/transparent regions unrelated to the run being filled (e.g. background visible on both sides of a small removed-object hole, elsewhere on the same line), and using the line's extremes instead of the run's immediate neighbors would source colors from those unrelated regions instead of the pixels actually adjacent to the gap. A run missing a bordering non-transparent pixel on either side (including running off the image edge before finding one) is left untouched.

Rather than using that single immediate bordering pixel's color directly (which tumbles hard if it happens to be an outlier — a stray speck of noise sitting right at the gap's edge), `_anchor_run_average` (a private helper) continues walking further along the line, away from the gap, from each immediate bordering pixel: it keeps collecting further pixels as long as they're contiguous (a transparent pixel immediately ends the walk on that side — no bridging back across a second gap) and within `tolerance` of the *immediate bordering pixel's* color specifically (not neighbor-chained the way `gradient_select`'s walk is — every pixel in the run is compared against the same fixed reference, so a slow drift across many similar pixels can't gradually pull the average away from what the gap's edge actually looks like). The mean RGB color of everything collected this way (at least the bordering pixel itself, if nothing further qualifies) is that side's actual anchor color for the interpolation below — this is the fuzzy-select-style tolerance test applied along a line instead of flood-filled across a 2D area, and it's what smooths over a single outlier pixel instead of letting it dictate an entire gap's fill color. `tolerance` is a plain `int` request param (default 32, same shape/default as `fuzzy_select`'s), read from the same Threshold tool-options value the browser already persists for Fuzzy Select/Gradient Select/Select by Color rather than introducing a fourth stored preference.

Every pixel in a valid run then gets its RGB linearly interpolated between its two *averaged* anchor colors by its normalized position within the run (the interpolation's start/end position along the line is still the immediate bordering pixel's position, only the color at that position is now an average rather than that one pixel's own color), and its alpha set to opaque (255); anchor colors can come from *outside* the selection (e.g. background texture just past the selection's edge) while the write itself stays restricted to selected-and-transparent pixels — the tool is for blending across a gap (a scratch, a removed object) along the grain of a gradient or textured background, not a generic fill. Unlike `fuzzy_select`/`invert`/every mask-producing endpoint, this returns `{png}` (a full RGBA image), the same response shape as `delete` — the Rails side composites it as a new `canvas_image` snapshot rather than a new mask (see below), since it changes what the canvas *shows*, not what's *selected*.

On the Rails side, `EditSessions::SelectionsController#fuzzy_select` takes additional `add`/`subtract` params (from Shift+click / Ctrl/Cmd+click, see §5), handled by the shared `mask_with_modifier` helper: when a mask is already attached and `subtract=true`, it calls `fuzzy_select` for a fresh single-region mask and removes it from the current mask via `/subtract`; when `add=true` instead, it unions the fresh region into the current mask via `/combine`, exactly as before. The two params are mutually exclusive (subtract is checked first); with neither set, or with no current mask, the fresh region simply replaces the selection. `#free_select`, `#line_select`, `#rect_select`, and `#brush_select` take the same `add`/`subtract` params and route through the same `mask_with_modifier` helper — closing a free-select path, placing line select's or rect select's second point, or releasing a brush stroke, with Shift or Ctrl/Cmd held is the equivalent of the same modifiers on fuzzy select.

All Rails-side calls to these endpoints go through a single client class (`PixelEngineClient`, see §5) — no controller or model calls the Python service directly.

## 5. Rails app structure

- `app/models/edit_session.rb` — the state owner (original image + current mask + accumulating canvas image + undo/redo snapshot history/position).
- `app/models/edit_session_snapshot.rb` — one undo/redo step; a pure history record (mask + canvas_image attachments, a position, and an operation label). Never read or written by controllers directly — only through `EditSession#record_snapshot!`/`#undo!`/`#redo!`, so `EditSession` stays the sole state owner per §3.
- `app/controllers/edit_sessions_controller.rb` — `new`/`create` on upload, `show` renders the editor, `export` streams `EditSession#display_image` (see §3) as a download.
- `app/controllers/edit_sessions/selections_controller.rb` (`EditSessions::SelectionsController`, namespaced under `edit_sessions` to match the nested routes) — the tool actions (`fuzzy_select`, `gradient_select`, `select_by_color`, `gradient_select_from_selection`, `select_all`, `deselect_all`, `free_select`, `line_select`, `brush_select`, `rect_select`, `split_selection`, `smooth_auto_fill`, `invert`, `remove_holes`, `grow_selection`, `select_border`, `delete`, `undo`, `redo`, `save_selection`, `saved_selections`, `merge_selection`). Every action except `deselect_all` and `undo`/`redo` calls `PixelEngineClient` for pixel math and then `EditSession#record_snapshot!` to persist the result as a new undo step; `deselect_all` records a snapshot with `mask_bytes: nil` (no pixel math needed to drop a selection); `undo`/`redo` just call the corresponding `EditSession` method. `remove_holes` follows the exact same shape as `invert` (`render_no_selection` guard, single mask in/out through `PixelEngineClient`, one snapshot). `grow_selection` and `select_border` follow that same shape, just also requiring a `border_size` param (from the shared Border Size dialog, see below) alongside the current mask. `split_selection` also follows that `render_no_selection`-guarded, single-mask-in/mask-out shape, taking `x1`/`y1`/`x2`/`y2`/`keep_x`/`keep_y` params instead of a `border_size` — same family as `invert`/`grow_selection`/`select_border` (a one-shot transform of the current selection with no add/subtract modifier), not the `mask_with_modifier` family `fuzzy_select`/`free_select`/`line_select` belong to. `gradient_select` and `select_by_color` are both identical in shape to `fuzzy_select` — same params, same `mask_with_modifier` call for add/subtract — just calling `PixelEngineClient#gradient_select`/`#select_by_color` instead. `gradient_select_from_selection` also follows the `render_no_selection`-guarded, single-mask-in/mask-out shape (like `grow_selection`), taking the current mask plus a `tolerance` param (reusing the same Threshold value `fuzzy_select`/`gradient_select` read) instead of a seed point — see §1/§4 for why it needs its own endpoint rather than reusing `gradient_select`. `line_select` follows the same `mask_with_modifier` shape as `fuzzy_select`/`free_select`, just taking `x1`/`y1`/`x2`/`y2`/`brush_size` params instead of a seed point or point list. `rect_select` follows that identical shape too, just without `brush_size` — the two opposite corners `x1`/`y1`/`x2`/`y2` are all it needs. `brush_select` also follows that `mask_with_modifier` shape, taking a `strokes` param (an array of point arrays, one per client-side drag — see §5's `editor_controller.js` note) plus `brush_size` instead of a seed point/segment/polygon. `smooth_auto_fill` guards on `render_no_selection` like `invert`/`split_selection`, but otherwise follows `delete`'s shape rather than any mask-producing action's: it downloads `display_image` and `current_mask`, calls `PixelEngineClient#smooth_auto_fill` with the `x1`/`y1`/`x2`/`y2` direction-vector params plus `tolerance` (`params.fetch(:tolerance, 32).to_i`, same default/shape as `fuzzy_select`'s), and calls `record_snapshot!` with `canvas_bytes` set to the composited PNG and `mask_bytes` carrying the current mask forward unchanged (the selection itself never changes) — responding with `{final_image_url}`, not `{mask_url}`, the same response shape `delete` uses. `save_selection` guards on `render_no_selection` like `invert`, then calls `EditSession#save_selection!` — no `PixelEngineClient` call and no snapshot, since saving doesn't change what's selected (see §3). `saved_selections` (a `GET`, unlike every other action here) and the shared `render_saved_selections` private helper both just list the session's saved selections as `{id, slot, mask_url}` JSON, no pixel math. `merge_selection` looks up the chosen `EditSessionSavedSelection` by `saved_selection_id` and delegates to `EditSession#merge_selection!`, which itself calls `PixelEngineClient#combine` — same op `mask_with_modifier`'s add branch uses — and `record_snapshot!`. Each responds with JSON (`{mask_url}` and/or `{final_image_url}`) for the Stimulus controller to redraw.
- `app/services/pixel_engine_client.rb` — the **only** boundary that talks to the Python service. Thin `Net::HTTP` wrapper (base64-encodes/decodes at the edges, no business logic). This is what makes the engine swappable later (e.g. adding an ML-based endpoint) without touching controllers.
- `app/javascript/controllers/editor_controller.js` — Stimulus controller owning canvas rendering, click handling, keyboard shortcuts, and tool-selection UI state. Contains no pixel math — it sends tool + coordinates to Rails and redraws whatever mask/image comes back. Renders the selection as a GIMP-style "marching ants" dashed boundary outline (black + white, offset out of phase) on a second `<canvas>` stacked on top of the image canvas — no tint over the selected pixels, so the image underneath stays fully visible. The dash offset scrolls continuously via `requestAnimationFrame` (`march()`), rather than the stroke alternating color, since a moving dash reads more clearly than a static or blinking one. Undo/redo (`undo()`/`redo()`) share a `stepHistory` helper that redraws the base canvas and either redraws the returned mask or clears the overlay if the position has none. Zoom (`setZoom()`) applies a CSS `transform: scale()` to `.canvas-stack` (the shared positioning parent of both canvases, `data-editor-target="canvasStack"`) rather than resizing either `<canvas>` element — the canvases' `width`/`height` attributes (and therefore their backing pixel data) never change, so zoom is purely a display-layer concern. 100% zoom means true 1:1 pixel display (no `max-width` cap on the canvas elements), matching GIMP/Photoshop convention; large images may exceed the visible viewport at 100%+ and require scrolling `.editor-main` (which has `overflow: auto`) to see the rest — this is expected, not a bug. `.canvas-stack` uses `margin: auto` (not flexbox `align-items`/`justify-content: center`) to center within `.editor-main`, since centering via flex alignment breaks the ability to scroll to the start edge of overflowing content in all major browsers. Click-to-pixel coordinate mapping for fuzzy select already derives from `overlayCanvasTarget.getBoundingClientRect()` scaled against the canvas's pixel dimensions, which transparently accounts for CSS transforms, so zoom required no changes there. `eventToImageCoordinates()` clamps its result to `[0, width - 1]`/`[0, height - 1]` before returning, GIMP-style: a click just outside the image lands on the nearest edge pixel instead of producing an out-of-bounds coordinate, so every tool sharing this one chokepoint — free select's vertices, line select/rect select/split selection's points, brush strokes — can be placed without having to pixel-perfectly hit the image boundary. This only helps once a click actually reaches the handler, though, so the `click`/`mousemove` listeners are bound to `.editor-main` (the padded scroll container, `data-editor-target="editorMain"`), not `overlayCanvas` itself — the canvas element's own box ends exactly at the image edge, so binding there would mean clicks landing in `.editor-main`'s padding never fire at all, clamping notwithstanding. The crosshair/zoom-in cursor (`.editor-main`/`.editor-main.zoom-cursor`, toggled by `updateToolOptions()`) moved to the same element for the same reason — cursor and click target must agree so hovering near the edge doesn't show a crosshair that then does nothing on click. Brush's `mousedown`/`mouseleave` stay bound to `overlayCanvas` itself (not moved to `.editor-main`): a brush stroke paints pixels, so there's nothing meaningful to start painting outside the image, unlike a free-select vertex or line-select/rect-select endpoint, which are just points that get clamped inward. Gradient Select (`this.tool === "gradient_select"`) and Select by Color (`this.tool === "select_by_color"`) both reuse fuzzy select's branch of `click()` entirely — same coordinate mapping, same `add`/`subtract` modifier handling, same request body shape — the only difference is `selectUrlForTool()` picks `gradientSelectUrlValue`/`selectByColorUrlValue`/`fuzzySelectUrlValue` based on `this.tool`, since all three tools are otherwise identical from the browser's perspective (see §4 for the Python-side algorithm differences). Select by Color also shares the Threshold tool option with Fuzzy Select/Gradient Select (`updateToolOptions()` shows `toleranceControl` for all three), reading and writing the same persisted value. Gradient Select from Selection (`gradientSelectFromSelection()`) is not a toolbox tool and has no click handling of its own — like `removeHoles()`/`growSelection()`, it's a one-shot Functions-row action guarded by `this.selectionPath`, POSTing just `{ tolerance: this.tolerance }` to `gradientSelectFromSelectionUrlValue` and redrawing the returned mask. It deliberately reuses `this.tolerance` (the same Threshold value the toolbox's Gradient Select reads) rather than introducing a second stored preference, since conceptually it's "resume the gradient walk at the same tolerance," not a distinct tool option. The Zoom tool shares the same `click()` entry point as fuzzy select — `click()` dispatches on `this.tool` first, so exactly one canvas click handler exists rather than a second one bolted on. `zoomAtEvent()` computes the click position relative to `.editor-main` (the scroll container, `data-editor-target="editorMain"`) in *content* coordinates (`scrollLeft/Top` + position within the viewport) before the zoom change, then after `setZoom()` rescales, sets `.editor-main`'s `scrollLeft`/`scrollTop` so that same content coordinate — scaled by the new/old zoom ratio — lands back under the same viewport position. This is what keeps the clicked image pixel visually stationary under the cursor (GIMP's "zoom to point" behavior) rather than always zooming from the top-left corner. Keyboard zoom (`+`/`-`/`0`) has no click point, so it anchors on `viewportCenter()` instead — the middle of whatever is currently visible in `.editor-main`. Free select (`this.tool === "free_select"`) reuses the same `click()` entry point and the same `eventToImageCoordinates()` pixel-coordinate mapping fuzzy select uses, but instead of firing a request per click, `freeSelectClick()` accumulates vertices into `this.freeSelectPoints` client-side and only POSTs once the path closes. A path closes when a click lands within `FREE_SELECT_HANDLE_RADIUS` image pixels of the first vertex (`isNearFirstPoint()`) and at least 3 vertices exist; `submitFreeSelect()` then sends the full point list in one request (Shift+click on the closing click sets `add: true`, same modifier convention as fuzzy select). `renderOverlay()` draws the in-progress path (plain solid line, no marching ants — that animation is reserved for a committed selection) on top of whatever committed selection outline is already showing, since starting a new lasso doesn't clear the existing selection until the path actually closes; `renderFreeSelectPath()` also draws the enlarged first-point handle circle. `cancelFreeSelect()` (bound to `Escape`, and called automatically on tool switch) clears `freeSelectPoints` without submitting anything. Because `points` is an array of `{x, y}` objects, `post()` no longer builds its request body with a bare `URLSearchParams(params)` — `appendFormParams()` recursively bracket-encodes nested arrays/objects (`points[][x]=…&points[][y]=…`) the way Rails' form param parser expects, since `URLSearchParams` itself only understands flat key/value pairs. An array of arrays (Brush's `strokes`, see below) needs the outer level explicitly indexed (`strokes[0][]`, `strokes[1][]`, ...) rather than doubly-bracketed (`strokes[][]`) — Rack's parser can't tell where one `[]`-nested array ends and the next begins when both levels are unindexed, and silently collapses every stroke's points into just the last stroke. `appendFormParams()` detects this case (an array whose items are themselves arrays) and indexes only that level; every other array this app sends (flat arrays of scalars or `{x, y}` objects) is unambiguous and keeps the plain `[]` encoding. Rails parses an indexed outer level as an `ActionController::Parameters` hash keyed by string index rather than an actual array, so `EditSessions::SelectionsController#brush_select` calls `.values` on it to recover stroke order (see §4/§5).

Line select (`this.tool === "line_select"`) also reuses the same `click()` entry point and `eventToImageCoordinates()` mapping, but unlike free select's multi-vertex path it only ever needs two points: the first click stores `this.lineSelectStart` and returns without a request; the second click (`lineSelectClick()`) sends both points plus `brush_size: this.brushSize` in one request via `submitLineSelect()`, reading the Shift/Ctrl/Cmd modifier keys off that second click exactly the way `freeSelectClick()`'s closing click does. A `mousemove` listener on `.editor-main` (`pointerMove()`, bound/unbound in `connect()`/`disconnect()` alongside the existing keydown listener) tracks `this.lineSelectPointer` only while a first point is pending, so `renderLineSelectPreview()` can draw a live preview segment from the placed first point to the current pointer position, stroked at `this.brushSize` width — this is what lets the user see exactly what a given Brush Size will select before committing the second click. `cancelLineSelect()` (bound to `Escape`, and called automatically on tool switch, mirroring `cancelFreeSelect()`) clears `lineSelectStart`/`lineSelectPointer` without submitting anything. Brush Size (`restoreBrushSize()`/`setBrushSize()`) follows the exact same stepper+slider+`localStorage` pattern as Fuzzy Select's Threshold (`BRUSH_SIZE_STORAGE_KEY`, `.tool-option-row[data-editor-target="brushSizeControl"]`), just with integer steps and a 1–200 range instead of Threshold's 0–128 decimal range. `updateToolOptions()` shows that same row for both Line Select and the Brush tool, since they share one Brush Size preference rather than each getting its own.

Rectangle select (`this.tool === "rect_select"`) reuses the exact same two-click shape as Line Select — `rectSelectClick()`/`submitRectSelect()` mirror `lineSelectClick()`/`submitLineSelect()` structurally — but has no Brush Size or any other tool option, since `rect_select` fills the whole rectangle rather than stroking a path through it (see §4); `updateToolOptions()` doesn't show either tool-options row for it, the same "no tool options" treatment as Free Select and Zoom. `pointerMove()` tracks `this.rectSelectPointer` the same way it tracks `this.lineSelectPointer`, so `renderRectSelectPreview()` can draw a dashed, translucent-filled rectangle from the placed first corner to the current pointer position — the rectangle preview's fill (not just an outline) is what makes it read as "this whole area gets selected" rather than "this diagonal gets stroked," the visual distinction from Line Select's preview. `cancelRectSelect()` (bound to `Escape`, and called automatically on tool switch, mirroring `cancelLineSelect()`) clears `rectSelectStart`/`rectSelectPointer` without submitting anything.

The Brush tool (`this.tool === "brush"`) is the one tool whose input is a continuous drag rather than a discrete click, so it does not go through `click()`/`clickNow()` at all (which no-ops immediately when `this.tool === "brush"`) — instead `connect()` binds dedicated `mousedown`/`mouseup`/`mouseleave` listeners (`brushPointerDown()`/`brushPointerUp()`/`brushPointerLeave()`) alongside the existing `pointerMove()` mousemove listener, mirroring the existing bind/unbind-in-connect/disconnect pattern. `brushPointerDown()` starts a new point array and pushes it onto `this.brushStrokes` (an array of strokes, not a single flattened point list — see §4 for why `masks.brush_select` needs strokes kept separate; each commit is typically one stroke from one drag); `pointerMove()` appends to that array while a drag is in progress; `mouseup` (bound on `window`, not the canvas, so releasing outside the canvas still ends the drag) commits via `submitBrush()`, reading Shift/Ctrl/Cmd off that mouseup exactly the way free select's closing click and line/rect select's second click do. `mouseleave` only clears the brush-size cursor preview — it does not end or commit the stroke, so leaving mid-drag and releasing outside still selects what was painted. None of the painting touches Rails until that mouseup, which is what makes brush drawing not round-trip per frame the way every other tool's discrete click does.

Rendering is a third canvas, `brushCanvasTarget` (`data-editor-target="brushCanvas"`), stacked between `imageCanvas` and the marching-ants `overlayCanvas` in the DOM (and therefore in paint order, both being `position: absolute` with no explicit `z-index`) — kept as its own layer rather than drawn into `overlayCanvas` so `renderOverlay()`'s clear-and-redraw of the selection outline never has to know about or preserve in-progress brush painting, and vice versa. `renderBrushCanvas()` draws each `this.brushStrokes` entry as a translucent blue rounded-join polyline at Brush Size width (`drawBrushStroke()`, or a single filled circle for a one-point stroke/click-with-no-drag) — the same preview language as Line Select's live segment, with no full-canvas tint. It also draws a live circle at `this.brushPointer` sized to the current Brush Size when not actively dragging, as a brush-size cursor preview. `updateToolOptions()` calls `renderBrushCanvas()` when switching into the brush tool and `clearBrushCanvas()` when switching out of it; `setBrushSize()` also re-renders it live so the cursor preview and any future strokes reflect the new size immediately.

`submitBrush({ add, subtract })` (enqueued like every other selection-mutating action) snapshots `this.brushStrokes`, clears the local painting state and canvas, and POSTs `{ strokes, brush_size: this.brushSize, add, subtract }` to `brushSelectUrlValue` — the same `{mask_url}` response shape every other select tool redraws via `drawMask()`. `cancelBrush()` (bound to `Escape`, called automatically on tool switch away from brush, and also called from `stepHistory()` so an in-progress brush painting doesn't survive an undo/redo) discards `this.brushStrokes` without submitting anything, the same convention as `cancelFreeSelect()`/`cancelLineSelect()`/`cancelSplitSelection()`.

Split Selection (`this.tool === "split_selection"`) reuses the same `click()` entry point as every other tool but needs three clicks, not one or two: the first two (`splitSelectionClick()`, accumulated into `this.splitSelectionLine`) place the infinite line's direction points, and the third submits immediately via `submitSplitSelection()` — no closing gesture like Free Select's, and no add/subtract modifier like Line Select's second click, since Split Selection replaces the current selection rather than adding to it (guarded by `if (!this.selectionPath) return` at the top of `splitSelectionClick()`, the same silent-no-op convention as `invertNow()`/`removeHolesNow()`, rather than disabling the toolbar button — see the "requires a selection" note in the button's `title`). `renderSplitSelectionPreview()` draws the two placed points, then — once both are placed — the line extended to the full canvas bounds (`clipExtendedLine()`, a manual line-vs-rectangle clip since this needs the two boundary-crossing points as data, not just a canvas draw call) rather than just the short segment between the two clicks, since the segment itself is not what gets used server-side; once the pointer moves after that, a translucent fill over whichever half the pointer is currently over previews what a third click there would keep (`tracePositiveSidePolygon()`), using the same sign-of-cross-product side test as `masks.split_selection` so the preview never disagrees with what the server will actually compute. That polygon has to walk the rectangle's corners in the order encountered along its perimeter between the line's two boundary-crossing points, not a fixed corner order — a fixed order self-intersects ("bowties," filling nothing) whenever the kept corners straddle the perimeter's fixed start point, which is a completely ordinary line orientation, not an edge case. `cancelSplitSelection()` (bound to `Escape`, and called automatically on tool switch, mirroring `cancelFreeSelect()`/`cancelLineSelect()`) clears `splitSelectionLine`/`splitSelectionPointer` without submitting anything.

Smooth Auto Fill (`this.tool === "smooth_auto_fill"`) reuses the same `click()` entry point but needs only two clicks, like Line Select — except, like Split Selection's two points, they set direction/position only, not a stroke's endpoints, and the tool requires an existing selection (guarded by `if (!this.selectionPath) return` at the top of `smoothAutoFillClick()`, the same silent-no-op convention as `splitSelectionClick()`/`invertNow()`). The second click submits immediately via `submitSmoothAutoFill()`, sending `tolerance: this.tolerance` alongside the direction-vector params (same shared Threshold value Fuzzy Select/Gradient Select/Select by Color read/persist — `updateToolOptions()` shows `toleranceControl` for Smooth Auto Fill too, rather than introducing a separate tool option), with no add/subtract modifier — like Split Selection, this doesn't change what's selected, only pixel colors within it. `renderSmoothAutoFillPreview()` draws a plain line from the placed first point to the current pointer position (tracked the same way `lineSelectPointer`/`rectSelectPointer` are, via `pointerMove()`), at a fixed 1px width rather than Brush Size, since the line here only conveys direction, not a stroke width that determines what gets selected. Because the response is `{final_image_url}`, not `{mask_url}` (see §4 — this tool writes pixel colors like `delete`, not a mask), `submitSmoothAutoFill()` calls `redrawCanvas()` the same way `clearSelectionNow()`/`stepHistory()` do, rather than `drawMask()`; the selection outline itself is left untouched since the mask doesn't change. `cancelSmoothAutoFill()` (bound to `Escape`, called automatically on tool switch, and also cleared in `stepHistory()` alongside Line/Rect Select's in-progress state) clears `smoothAutoFillStart`/`smoothAutoFillPointer` without submitting anything, mirroring `cancelSplitSelection()`/`cancelLineSelect()`.

Grow Selection and Select Border (`openGrowSelectionDialog()`/`openSelectBorderDialog()`, Functions-row buttons only, no toolbar entry or keybinding) both prompt for a Border Size via one shared `<dialog>` (`data-editor-target="borderSizeDialog"`) rather than each getting its own — matching GIMP's own single "border size" convention across the two operations. Opening either sets `this.borderSizeAction` to remember which request to fire, pre-fills the dialog's stepper input from `this.borderSize`, and calls `showModal()`; `confirmBorderSizeDialog()` (the dialog's Apply button) persists whatever value is currently in the input via `setBorderSize()` before closing the dialog and dispatching to `growSelection()` or `selectBorder()` based on `this.borderSizeAction` — both of those just POST `{border_size: this.borderSize}` and redraw the returned mask, the same one-shot mask-in/mask-out shape as `invert()`/`removeHoles()`. `closeBorderSizeDialog()` (Cancel button or the dialog's close button) just closes without dispatching anything. Border Size itself (`restoreBorderSize()`/`setBorderSize()`) follows the same stepper+`localStorage` pattern as Threshold/Brush Size, but with no linked slider (`BORDER_SIZE_STORAGE_KEY`, 1–500 integer range) — since the dialog is a one-shot prompt rather than a persistent tool-options panel, a slider adds little.

Save Selection (`saveSelection()`, Functions-row button only, no dialog) just POSTs to `saveSelectionUrlValue` and reports success/failure via the status bar — there's nothing to choose, so unlike Grow/Border it needs no dialog at all. Merge Saved Selection (`openMergeSelectionDialog()`) fetches `savedSelectionsUrlValue` (JSON, not an HTML fragment — see below) and renders one row per saved selection into `<dialog data-editor-target="mergeSelectionDialog">`'s `data-editor-target="savedSelectionList"` `<ul>` via `renderSavedSelectionList()`, each row built with a couple of `document.createElement` calls (thumbnail `<img>`, slot label, a "Merge" button) rather than an HTML template, since there's no server-rendered partial in play here the way `_session_list.html.erb` backs the Load Session dialog. Clicking a row's Merge button calls `mergeSelection(savedSelectionId)`, which closes the dialog and POSTs `{saved_selection_id}` to `mergeSelectionUrlValue`, redrawing the returned mask the same one-shot way as every other select op. `closeMergeSelectionDialog()` mirrors `closeBorderSizeDialog()`.

`click()` and `freeSelectClick()`/`submitFreeSelect()` read `event.shiftKey` as `add` and `event.metaKey || event.ctrlKey` as `subtract` on the same request — GIMP's mapping (Shift = add, Ctrl = subtract). The two are mutually exclusive at the Rails layer (see §4); the browser doesn't need to enforce that itself since it just forwards whichever modifier keys were actually held.

Entries: Fuzzy Select, Free Select, Gradient Select, Select by Color, Line Select, Rectangle Select, Brush, Split Selection, Smooth Auto Fill, Zoom. Free Select (`data-tool="free_select"`) follows the same pattern. Brush (`data-tool="brush"`) is the one toolbox entry that isn't click-driven at all — it owns dedicated mousedown/mousemove/mouseup listeners instead (see §5's `editor_controller.js` note) — but it sits in the toolbox for the same reason as Free Select/Line Select/Rectangle Select: it's a distinct input mode that owns what interacting with the canvas does, not a one-shot Functions-row button. Laid out as `.function-button-list`, a `display: flex; flex-wrap: wrap` list, so it wraps to multiple rows within the sidebar's fixed width instead of needing a wider sidebar or a scrollable strip.

- **Status bar** (`.editor-statusbar`) — bottom strip showing the current operation status and a keybinding hint.

### Session management UI (upload + load, both pages)

`app/javascript/controllers/global_actions_controller.js` is mounted on `<body>` in `app/views/layouts/application.html.erb` (the one layout shared by both the landing page and the editor page), rather than on a per-page element, specifically so upload/load behavior is identical everywhere without duplicating markup or JS per page. It owns:

- A hidden, always-present `<form>` (in the layout) wrapping the real `edit_session[original_image]` file input used by `EditSessionsController#create` — both the native file-picker path and drag-and-drop assign the chosen file(s) to that input (drag-and-drop via `DataTransfer`, the standard way to programmatically populate a file input) and call `form.requestSubmit()`. This keeps upload a plain full-page form POST/redirect, unchanged from `create`'s original behavior — no JSON response format, no fetch-based upload.
- Page-wide (`document`-level) `dragover`/`drop` listeners (no visible drop-target UI — dropping anywhere on either page works) assign the dropped file to the hidden form's input and submit it, the same path the file picker uses.
- The Load Session `<dialog>` (also always present in the layout): `Ctrl/Cmd+O` or a "Load Session" button fetches `EditSessionsController#index`, which renders `app/views/edit_sessions/_session_list.html.erb` as a bare HTML fragment (`layout: false`) — injected via `innerHTML`, matching this app's existing plain-fetch-plus-DOM-update style (there is no Turbo Frames/Streams usage anywhere in this app, despite `turbo-rails` being in the Gemfile as Rails 8's default). Each row's delete button calls `DELETE /edit_sessions/:id` behind a `confirm()` prompt, then removes just that row from the open dialog on success rather than re-fetching the whole list.
- The two keyboard shortcuts, `Ctrl/Cmd+Shift+O` (open file picker) and `Ctrl/Cmd+O` (open Load Session dialog), bound via the same manual `document.addEventListener("keydown", ...)` pattern `editor_controller.js` uses (not `data-action="keydown@window->..."`). Both controllers' keydown listeners coexist on the editor page without conflict: neither calls `stopPropagation()`, and the shortcut sets don't overlap (`editor_controller.js` claims I/A/D/Z/Shift+Z/Shift+H/S plus plain `U`/`F`/`G`/`O`/`L`/`B`/`S` for tool selection; `global_actions_controller.js` claims Ctrl/Cmd+O/Ctrl/Cmd+Shift+O only — the modifier is what keeps its `O` binding from colliding with the editor's plain, unmodified `O` for Select by Color).

## 6. Python service structure

Lives in its own subdirectory, `python_service/`, with its own virtualenv (`python_service/.venv`) and dependencies pinned in both `pyproject.toml` and `requirements.txt`. Two modules: `app.py` (FastAPI HTTP layer — base64 in/out, no pixel math) and `masks.py` (OpenCV/Pillow pixel math — stateless functions, no HTTP or persistence concerns).

Start command (development): `cd python_service && source .venv/bin/activate && uvicorn app:app --port 5001 --reload`. Never imported into the Rails process; the only coupling is the HTTP contract in §4. `PixelEngineClient` defaults to `http://localhost:5001`, overridable via `PIXEL_ENGINE_URL`.

`bin/dev` starts both Rails and the Python service together via `foreman` (declared in the Gemfile's `:development` group) reading `Procfile.dev` at the repo root:

```
web: bin/rails server
python: cd python_service && .venv/bin/uvicorn app:app --port 5001 --reload
```

`--reload` is always on for the Python process in development so `masks.py`/`app.py` changes pick up without a manual restart, matching Rails' own code-reloading behavior. This is the one command to run both services; there is no separate "start Python manually" step for day-to-day development.

## 7. Output contract & future integration

The exported file is a transparent PNG (RGBA), dimensions unchanged from the source. This intentionally matches `fang-backend`'s `Sprite` model, which stores `url` (a path to a pre-made transparent PNG/WebP) plus optional `x`, `y`, `width`, `height` for display positioning. There is **no live integration today** — this tool only produces a downloadable file — but keeping the output format aligned means a future task can wire uploads directly into `fang-backend` with minimal rework.

If ML-based one-click background removal is added later (e.g. via `rembg`), it plugs in as a new Python endpoint plus a new Rails service method — see §8. It does not change the architecture in §2.

## 8. How to add a new tool

1. Add the new operation's endpoint to the Python service (stateless: takes image/mask + params, returns a new mask or final PNG).
2. Add a corresponding method to `app/services/pixel_engine_client.rb`.
3. Add the controller action (in `selections_controller.rb` or a new controller if it's not a selection op) that calls the client and persists the result via `EditSession#record_snapshot!` (see §3) so the operation participates in undo/redo — don't attach directly to `current_mask`/`canvas_image`.
4. Add the toolbar button + Stimulus action in `editor_controller.js` to trigger it and redraw the returned mask/image.
5. Update the tool list in §1 and, if the contract changed, the table in §4.

No other files should need touching for a same-shape tool addition (i.e. one that fits the existing mask-in/mask-out or mask-in/PNG-out request shape).
